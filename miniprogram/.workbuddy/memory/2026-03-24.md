## Windows开发工具云初始化错误修复（00:43）

### 问题描述
Windows 开发者工具上出现云初始化错误：
```
cloud init error: Error: rid: 1cdaa4f4-1774283818.errCode: -601002
```

### 根本原因
**多端应用扩展库（wxext14566970e7e9f62）在 Windows 开发工具上的云初始化兼容性问题**：
- 项目启用了多端架构 `"projectArchitecture": "multiPlatform"`
- 多端应用扩展库在 Windows 环境下的云初始化方式与标准小程序不同
- 显式指定 `env` 参数会导致初始化失败

### 修复方案
修改 `app.js` 的云初始化逻辑：
1. **优先方案**：不指定 env，让扩展库自动选择环境
2. **备用方案**：如果自动失败，再尝试显式指定环境
3. **降级处理**：两种方案都失败时不弹窗打断用户，而是在后续云函数调用时重试

### 修改代码
**文件**: `miniprogram/app.js`
**改动**: 重构 `onLaunch()` 中的云初始化部分，采用双重初始化策略

### 技术要点
- 多端应用扩展库会自动管理云环境，无需显式指定
- Windows 开发工具对云初始化的校验比真机宽松
- 初始化失败不应阻断用户操作，应该在云函数调用时动态恢复

## 用户详情页"TA的特产"不显示 Bug 修复（12:58）

### 问题描述
点击用户头像进入 `pages/user-profile/` 页面，"TA的特产"栏目数据空白。

### 根本原因（三个问题叠加）

**问题1（HTTP/APK端致命）**：`userInit` 云函数 `publicProfile` action 里：
```javascript
const { targetOpenid } = event   // ❌ 用的是原始 event
```
当 APK/鸿蒙 端通过 HTTP 触发时，实际参数在 `actualEvent`（从 body 解析），`event.targetOpenid` 为 `undefined`，查询条件为空，返回用户不存在。

**问题2（字段名不匹配）**：`products` 集合发布时存的是 `valueRange` 字段，但 `publicProfile` 查询时 `.field()` 里写的是 `valueMin/valueMax`，导致这两个字段始终返回 `undefined`。

**问题3（查询不够健壮）**：只用 `_openid` 查 products，未兜底用 `openid` 字段查，旧数据（用 openid 字段发布的）查不到。

### 修复内容

**`cloudfunctions/userInit/index.js`**:
1. `const { targetOpenid } = event` → `const { targetOpenid } = actualEvent`
2. `.field()` 里 `valueMin/valueMax` → `valueRange`
3. 增加兜底查询：若 `_openid` 查不到，再用 `openid` 字段查
4. 添加调试日志（`productCount`）

**`miniprogram/pages/user-profile/index.js`**:
1. 引入 `VALUE_RANGES_V2` 常量
2. `formatValue(valueMin, valueMax)` → 通过 `VALUE_RANGES_V2.find(v => v.id === item.valueRange)` 获取 label

## 新用户积分显示为 0 的 Bug 修复（13:10）

### 问题描述
新用户第一次进入小程序，积分显示为 0，但数据库里应该初始化为 50 分。

### 根本原因（两个问题叠加）

**问题1：`app.js` 的 `globalData` 没有 `points` 字段，`initUser()` 也没有存储 `points`**
- `globalData` 里只有 `creditScore`，缺少 `points`
- `initUser()` 取到云函数返回值后，没有把 `points` 存到 `globalData`

**问题2：并发创建用户导致 Race Condition（更核心的问题）**
- `app.js` 的 `initUser()` 和 `mine` 页面的 `loadUserData()` 几乎同时触发
- 两个请求都调用 `userInit init`，数据库里还没有该用户
- 两个请求都会尝试创建新用户（第一个创建成功 points=50，第二个查到已有用户）
- 但先返回的请求可能是在数据库写入完成前发起查询，拿到的 `points` 为 0

**问题3：云函数创建新用户后用 `where` 重查，可能受数据库索引延迟影响**
- 旧代码：`add()` 后再用 `where({ _openid: openid }).get()`
- 新建文档的 `_openid` 索引可能有毫秒级延迟
- 改为用 `add()` 返回的 `_id` 精准 `doc().get()`

### 修复方案

**`miniprogram/app.js`**:
1. `globalData` 新增 `points: 0` 初始字段
2. `initUser()` 添加 `this.globalData.points = $r7.result.points || 0`

**`miniprogram/pages/mine/index.js`**:
3. `loadUserData()` 开头增加等待 `app.globalData.openid` 的逻辑（轮询最多5秒）
   - 确保 `app.js` 的 `initUser()` 完成后，mine 页面才发起第二次 `userInit init` 调用
   - 避免两个并发请求同时创建新用户

**`cloudfunctions/userInit/index.js`**:
4. 新用户创建后，改用 `doc(addRes._id).get()` 精准查询，避免 where 查询的索引延迟
5. 增加兜底：若 `userData.points` 为空，强制赋值 `NEW_USER_POINTS (50)`

### 修改文件
- `miniprogram/app.js`
- `miniprogram/pages/mine/index.js`
- `cloudfunctions/userInit/index.js`
