## 二、P0 功能详细设计

### 2.1 订单超时机制

#### 功能说明
- 确认后 **72小时** 未发货，自动取消订单
- 订单取消后，双方特产恢复为 `active` 状态
- 发送系统通知给双方

#### 数据结构
```javascript
// orders 表新增字段
{
  confirmTime: Date,        // 确认时间
  shipDeadline: Date,       // 发货截止时间（confirmTime + 72h）
  autoCancelReason: String, // 自动取消原因
}
```

#### 技术方案
**方案A：定时任务（推荐）**
- 云函数定时触发器，每小时执行一次
- 查询 `status: 'confirmed'` 且 `shipDeadline < now` 的订单
- 批量取消并释放特产

**方案B：延迟消息**
- 订单确认时设置延迟任务
- 到期自动触发取消逻辑

#### 开发工作量
| 任务 | 工作量 |
|------|--------|
| 云函数 `checkOrderTimeout` | 2h |
| 定时触发器配置 | 0.5h |
| 订单取消逻辑修改 | 1h |
| 前端超时提示 | 0.5h |
| **合计** | **4h** |

## 三、P1 功能详细设计

### 3.1 纠纷仲裁（人工介入）

#### 功能说明
- 用户可申请纠纷（已有 `DISPUTED` 状态）
- 管理员后台处理纠纷
- 支持查看证据、做出裁决

#### 现有状态
```
DISPUTED: 'disputed'  // 纠纷中
```

#### 新增功能
1. **前端**：订单详情页添加"申请纠纷"按钮
2. **前端**：纠纷申请表单（问题描述 + 图片证据）
3. **云函数**：`applyDispute` action
4. **管理后台**：纠纷列表、详情、裁决界面
5. **云函数**：`resolveDispute` action（管理员裁决）

#### 数据结构
```javascript
// orders 表新增字段
{
  dispute: {
    applicant: String,         // 申请人 openid
    reason: String,            // 纠纷原因
    images: Array,             // 证据图片
    applyTime: Date,           // 申请时间
    status: String,            // pending/resolved
    resolver: String,          // 处理人 openid
    result: String,            // 处理结果
    resultReason: String,      // 处理说明
    resolveTime: Date,         // 处理时间
  }
}
```

### 4.1 心愿单

#### 功能说明
用户发布想要换的特产，系统精准匹配并推送。

#### 数据结构
```javascript
// wishlist 表（新建）
{
  _id: String,
  _openid: String,
  province: String,          // 想要的省份
  category: String,          // 想要的品类
  keywords: Array,           // 关键词
  valueRange: String,        // 期望价值区间
  desc: String,              // 描述
  status: String,            // active/matched/removed
  matchedProducts: Array,    // 匹配到的特产ID
  createTime: Date,
}

// products 表新增字段
{
  matchedWishlistCount: Number,  // 匹配到的心愿单数
}
```

#### 匹配逻辑
```javascript
// 当有新特产发布时
async function matchWishlist(product) {
  const wishlists = await db.collection('wishlist').where({
    province: _.in([product.province, '']),  // 省份匹配或不限
    category: _.in([product.category, '']),  // 品类匹配或不限
    status: 'active'
  }).get()
  
  // 推送通知给匹配的用户
  for (const wish of wishlists.data) {
    await sendNotification(wish._openid, product)
  }
}
```

### 5.1 会员订阅

#### ⚠️ 重要提示
**会员订阅需要微信支付资质，且功能复杂，建议后期再做。**

#### 功能评估

| 能力 | 需要条件 | 当前状态 |
|------|----------|----------|
| 微信支付 | 商户号、资质认证 | ❌ 需申请 |
| 订阅消息 | 模板消息申请 | ❌ 需申请 |
| 会员体系 | 数据库设计 | ✅ 可扩展 |
| 权益系统 | 功能开发 | ❌ 待开发 |

#### 建议的会员权益（低复杂度版本）

| 权益 | 普通用户 | 会员用户 | 实现复杂度 |
|------|----------|----------|------------|
| 发布特产数量 | 5 个 | 20 个 | 低 |

| 优先匹配 | 普通 | 优先展示 | 中 |
| 每日签到积分 | +1 | +2 | 低 |
| 专属标识 | 无 | 会员徽章 | 低 |
| 去广告 | 有广告位 | 无广告 | 中 |

#### 开发工作量估算

| 任务 | 工作量 |
|------|--------|
| 会员数据表设计 | 2h |
| 会员状态管理 | 4h |

| 订阅管理 | 6h |

| 管理后台 | 4h |

#### 替代方案（先做权益，后做付费）

**第一阶段：积分会员（无付费）**
- 用户通过积分兑换会员权益
- 不涉及支付，降低风险
- 验证权益对用户的价值

**第二阶段：付费会员**
- 积累用户数据后，评估付费转化率
- 申请微信支付资质
- 开发订阅支付功能

## 七、风险与建议

### 风险点

1. **消息推送** - 需申请模板消息，审核可能不通过
2. **会员订阅** - 需微信支付资质，申请周期长
3. **防刷机制** - 邀请奖励可能被恶意刷取

### 建议

1. **先做低风险功能** - 订单超时、信用分、首次互换奖励
2. **同步申请资质** - 消息模板、微信支付
3. **灰度发布** - 新功能先小范围测试
4. **数据驱动** - 积累数据后再做付费会员
